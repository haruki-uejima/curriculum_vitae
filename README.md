## 複数の既存システムの改修、ITGC(IT統制)関連業務 【Ruby on Rails / TypeScript / React / AWS / Docker 】 (2022年02月〜)

【プロジェクト概要】複数の既存システム（AWSインフラを含む）の改修等

【担当業務】 複数の既存システムの改修、ITGC(IT統制)関連業務

- Ruby on Rails で開発された既存システムの改修
- 全てのバックエンドのAPIが React,TypeScript で開発された既存システムの改修（2022年05月〜）
- AWSインフラの整備（AWSリソースの整備、CloudWatchによるモニタリング整備等）
- ビジネスサイドからの機能追加・変更の要望への対応
- ITGC(IT統制)関連業務、IPOのためのワークフローの整備等を担当

#### 課題

- 以下のような課題がありました。
  - 新規開発の経験はあったが、既存システムの改修の経験がない。
  - 実務で Ruby on Rails で開発した経験がない。
  - React,TypeScript で開発した経験が全くない。
  - Web API に関する知識がない。

#### 取り組み

- こちらもAWSのインフラ構築時と同様に活用できるものはすべて活用するようにしました。Ruby on Rails と React,TypeScript は安価で良質な動画教材を購入してキャッチアップしました。Ruby on Rails と React,TypeScript のそれぞれに精通している先輩エンジニアにサポートも頂きながら少しづつ開発を進めることができるようになりました。
- 理解できないことがあってもすぐに質問するのではなくて、自力で調べても理解ができなければ質問するようにして、安易に先輩エンジニアの時間を奪うことのないように意識しました。その際に、Googleスプレッドシートを活用して内容整理をしたり、1人で考えすぎないように15分ルールを自分の中で決めて、日々の開発業務を進めています。
- 特に React,TypeScript で開発された既存システムについては、マイクロサービス、BFF を使用したシステム設計がされているので実装内容の理解に試行錯誤しています。
- ビジネスサイドからの既存システムへの改修依頼に対しての影響範囲調査、それに伴う仕様のドキュメント作成に関しては先輩エンジニアのドキュメントを見て学び、作成したものをレビューをして頂いてドキュメント作成スキルも磨いていきました。
- 現職はIPOに向けての準備をおこなっています。ITGC（IT統制）の関連で必要となるドキュメント類の作成や、ファイナンスグループ等の方々と連携を取りながら、IPOに向けての準備をおこなっています。また、全既存システムのAWSインフラの本番環境の EC2 や RDS のバックアップの証跡が必要であるにも関わらず、バックアップができていなかったり、開発環境のバックアップを取っていたりと細々したものの整備をする必要があったのでそういった業務も担当しています。
- Backlogを活用したワークフロー、CS（カスタマーサービス）等のビジネスサイドの方とコミュニケーションを取り、要望に応じた開発をおこないました。コミュニケーション取る際には既存システムはどのような仕様になっているか、要望に応じて改修をした場合にこういったところにも影響が及ぶことが想定される等の具体的な話をしながら連携を取っています。

#### 工夫した点等

- 既存システムはドキュメント類があまり作成されておらず、仕様を理解することに時間かかることもありましたがビジネスサイドの方とコミュニケーションを取って、既存システムを理解するように努めました。その際にはGoogleドキュメントや社内ドキュメントに内容を書き留めるようにして、社内ドキュメントとして残しました。
- Ruby on Rails と React,TypeScript 共にスキルがまだまだ未熟であり、継続的に勉強を進める必要があると理解していますので、勉強会に参加しています。その際にアドバイス頂いたことはドキュメントに書き留めています。Web API に関する知識も不足しているので動画教材・書籍を活用しています。量をこなして、スキルアップしていかなくてはいけいないので日々の積み重ねを大事にしています。

## AWSを使用したインフラ設計・構築 【AWS】 (2021年09月〜2022年01月)

【プロジェクト概要】 AWSを使用して、新システムのインフラを設計・構築

- 人手作業が多くかかっていた作業を自動化するためにシステム化しました。
- Wantedlyの[ストーリー](https://www.wantedly.com/id/haruki_uejima)に詳細な技術に関する内容等を書いています。

【担当業務】 システム全体のインフラ設計、インフラのコード化とデプロイの自動化

- AWSサポート等を有効に活用して適切なAWSサービスを選択
- AWSリソースの命名ルールを定めて、リソースを判別するための視認性の向上
- CloudFormationによるAWSのインフラ構築と管理
- CloudFrontとS3による静的リソース配信基盤の構築
- データベースには、Auora MySQL、アプリケーション実行環境には、ECS（Fargate）
- CI/CDのリリースフローには、CodePipeline（Blue/Green）、開発環境は CodePipeline（Rolling update）
- AWSのインフラ構成図等のドキュメント作成等

#### 課題

- 以下のような課題がありました。
  - 現職に入社後すぐにインフラ担当になったが、現職にはインフラ業務に精通しているエンジニアがいなかったため自力で進める必要があった。
  - 実務でAWSを使ったこと、インフラ業務の経験が全くなかった。
  - 独学で AWSの資格取得とAWSコンソールを使用したインフラ構築はしていたが AWS CloudFormation は使ったことがなかった。

#### 取り組み

- AWSでインフラ構築をするにあたり、マネージャーから AWSコンソール上からの作成しかできないのであれば IaCを利用せずにそちらでも良いというお話がありましたが、今後を見据えた場合 IaC化することのメリットがとても大きいと思い、 AWS CloudFormation を使ったインフラ構築をさせてくださいと直談判しました。（Terraform ではない理由は、AWSの標準形を採用すると社内で定められていたため）
- しかし、 AWS CloudFormationを使ったことがなかったので、初歩の基本的な知識から不足していたので苦労しました。AWSの動画学習サイトや書籍、AWS公式ドキュメントやAWSサポート等の活用できそうなものはすべて使って進めました。
- AWSに関する知識はもちろんのこと、CI/CD の知識等も不足していたのそれらの知識も補いながら進めました。期限があってのインフラ構築でしたので完璧を求めすぎずに、期限内に終えることを1番に考えつつ、周辺知識の理解もしながら進めるように意識しました。
- これらの取り組みによって、入社時点で決められていた期限までにインフラ構築をやりきることができました。業務の稼働時間は大幅に延びてしまいましたが、インフラ構築が遅延することによって、フロントエンド・バックエンドの開発メンバーに迷惑を掛けることなくやりきることができてほっとしましたし、達成感がありました。

#### 工夫した点等

- 独学でAWSの資格取得していたので、少しの知識はありました。それらの知識を都度思い返して、本当に今のインフラ構成で良いのかを考えるようにしていました。
- 自分の知識レベルで満足してしまってはいけないと考えていたので、定期的にAWSサポート等にインフラ構成についてのアドバイスを頂いていました。また、開発環境で構築した ECS（Fargete）の利用料金が大幅に上がってしまったことがあったのですが、その際にもアドバイス頂いてコストを最適化するように努めました。
- connpass の AWS CloudFormation の勉強会に参加させてもらったり、AWSの学習コミュニティの方々にアドバイスを求めたりもしていました。社内外の方にもサポートして頂けこともあり、新システムのインフラ構築業務をやりきることができました。今後の課題としては、既存システムのAWSのインフラの改修等に関するスキルが不足していることがあるので、AWSに関する自己研鑽は継続的におこなっています。

【その他】

- 開発プロジェクトは、スクラム開発で進めました。GoogleスプレッドシートとBacklogを活用して、プロダクトバックログとスプリントバックログを作成して進捗管理をおこないました。毎朝、朝会をして、Backlogのスプリントバックログに対しての進捗管理をおこない、チーム全体でサポートをしあいながら進めました。スプリントレビューとプランニングを実施して、ビジネスサイドの方への成果報告や進捗報告、次スプリントの計画をおこなっていました。レトロスペクティブでは、チーム・個人での KPTをおこなうことでメンバー同士で親交を深めつつ、プロジェクトをより良く進めるために役立ったと思います。
